### 1. hive-intel/crypto-mcp-onchain-dex

- **What:** A hosted MCP (Model Context Protocol) server exposing DEX analytics via HTTP. Five tools: pool info lookup, swap activity analysis, impermanent loss calculation, TVL aggregation, cross-DEX arbitrage opportunity scanning. No open-source code — the README points to an external hosted endpoint (`mcp.hiveintelligence.xyz`). Essentially an API wrapper product, not an algorithm library.
- **Key features:** Impermanent loss calculation for LP positions; yield farming APR/APY; slippage estimation; MEV detection (claimed but not described mechanically); cross-chain DEX aggregation; TVL tracking.
- **Gaps vs us:** (a) **Impermanent loss (IL) tracking for LP positions** — we track LP events but do not compute realized IL for an LP provider wallet. This would be valuable to custody and market-maker consumers who provide liquidity. (b) **Slippage estimation at the pool level** — we can derive this from pool reserves but do not emit it as a named signal. (c) **APR/APY computation on pool positions** — out of our anomaly-detection scope but useful to market-maker consumer.
- **Already have:** MEV detection is on our Phase 4 roadmap; swap monitoring via Raydium v4/CPMM adapter; pool TVL tracking via PoolEvent state.
- **Verdict:** Low research value. Closed-source hosted product; no novel detection techniques. IL computation is the one extractable idea, but it is a financial metric rather than an anomaly detector and belongs in the market-maker consumer, not in this service.

### 5. machenxi/rugpull-scam-token-detection (RugWatch)

- **What:** TypeScript Node.js tool that monitors new Solana token launches (Pump.fun, Raydium, Meteora) and emits alerts via Telegram/Discord. Implements a point-score risk model (0–100). Five risk checks: mint authority active (+30), freeze authority active (+20), insufficient initial liquidity <5 SOL (+15), unlocked LP (+20), unusual decimals (+5).
- **Key features:** New-token launch monitoring; mint/freeze authority checks; initial liquidity floor check; LP lock status check; decimal sanity check; weighted-penalty risk score (0–100); Telegram/Discord alert delivery.
- **Gaps vs us:** (a) **Initial liquidity floor check at launch** — we have D02 LP drain (ongoing) and D06 mint authority (ongoing), but we do not have a dedicated "is initial liquidity sufficient to constitute a real launch?" check. A token seeded with <5 SOL ($~750) is structurally de-risked to rug immediately. This is a distinct signal: a static snapshot check at first pool creation event, not a change-over-time detector. (b) **LP lock status at launch** — we have D02 Signal B which checks `lockers[]` from RugCheck, but we don't have a standalone check specifically for "LP is completely unlocked at launch time" as a severity signal. This overlaps D02 but at a different temporal point (instantaneous at launch vs. ongoing). (c) **Pump.fun new-token stream integration** — RugWatch explicitly monitors Pump.fun bonding-curve launches. We have Raydium v4 + CPMM adapters but Pump.fun bonding-curve token graduation events are a separate data stream. Graduating tokens (from bonding curve to Raydium pool) are a high-risk cohort.
- **Already have:** D01 honeypot (static checks include freeze authority); D02 LP drain (LP lock); D06 mint authority; scoring crate.
- **Verdict:** Moderately useful. The risk-point scoring methodology is shallow (no citations, no calibration corpus), but the signal checklist at launch time surfaces a genuine temporal gap in our detection: we run detectors on existing tokens but do not have a "first-N-minutes-of-life" profile scan. The Pump.fun graduation stream is the most actionable novel idea here.

### 10. 2654-zed/latent-flux

- **What:** Python system for L3 on-chain threat intelligence on Base, Arbitrum, and Optimism. Monitors smart contract deployments (not tokens) to detect malicious contracts: honeypots, Permit2 drainers, fee skimmers. Tracks deployer networks and maps criminal operations. Pre-transaction risk API for AI agents. Operationally live since March 2026. 124K contracts analyzed, 1.17M transaction events, 36K deployers.
- **Key features:**
  - **Contract-level threat classification** (not token-level): honeypot, Permit2 drainer, fee skimmer
  - **Deployer network graph** — maps relationships between deployers (organizational structure of criminal operations)
  - **Bayesian changepoint detection** (Adams & MacKay 2007) on transaction sequences to detect regime shifts in deployer behavior
  - **Tarjan SCC + Johnson cycle enumeration** on the address graph — detects circular fund flows structurally
  - **Unscented Kalman Filter** on transaction sequences for mean-reversion scoring
  - **Echo State Network (Reservoir Computing)** for multi-timescale deployer behavioral memory
  - **Multi-timescale consensus** (micro/meso/macro reservoirs, leak rates 0.3/0.05/0.01)
  - **Pre-transaction risk API**: risk_score, risk_tier, DO_NOT_APPROVE / CAUTION / PROCEED / UNVERIFIED
  - **Permit2 drainer detection** — specific to ERC-20 signature-based approval exploits
- **Gaps vs us:**
  (a) **Deployer network graph** — tracking that deployer A is organizationally linked to deployer B (shared funder, shared deployer EOA pattern) across multiple token contracts. We have `deployer_clusters` table in Postgres but the graph edges are not computed. This overlaps our Phase 3 graph/clustering work but latent-flux shows it is actionable today against real criminal operations.
  (b) **Bayesian changepoint detection on deployer behavior** — Adams & MacKay (2007) BOCPD is a rigorous, well-cited technique for detecting when a previously-normal deployer suddenly shifts behavior (starts rugging after a legitimate period). This is a temporal dimension we lack.
  (c) **Permit2 drainer detection (EVM)** — EIP-2612/Permit2 signature-based drainers are a distinct attack class on EVM chains where a contract tricks users into signing approvals that drain their wallets. Not applicable to Solana (different approval model), but firmly in Phase 4 EVM scope. Latent-flux has a working corpus.
  (d) **Tarjan SCC / Johnson cycle detection on the address graph** — cycle detection in the transfer graph is a stronger wash-trading signal than our current D05 Heuristic 1 (same-address round trips) + Signal B (cluster flow balance). A short cycle A→B→C→A in transfer graph = wash ring; Johnson's algorithm enumerates all elementary cycles. This is Phase 3 graph work.
  (e) **Pre-transaction risk API** — latent-flux produces a per-address pre-transaction risk score (DO_NOT_APPROVE / CAUTION / PROCEED). We produce per-token anomaly events but not per-address / per-contract pre-transaction advice. This API shape would be directly valuable to the trading bot consumer (before submitting a swap, check the contract) and to custody (before approving a withdrawal to an unknown address).
- **Already have:** Honeypot detection (D01 — token-level, not contract-level); deployer_clusters table (no edges computed yet).
- **Verdict:** The richest technical repo in the survey. Academically grounded (BOCPD is Adams & MacKay 2007, cited; SCC is Tarjan 1972). The deployer network graph + Bayesian changepoint detection combination is exactly the gap between our Phase 2 per-token detection and the Phase 3 graph layer. Permit2 drainer detection is a validated EVM Phase 4 item.

### Tier 1 — Ship this sprint (high ROI, <1 week effort, clear consumer value)

#### T1-1: New-token launch profile scan ("launch audit" detector, D08 candidate)

**Gap description:** We detect anomalies in tokens over time (change-based signals) but we have no "snapshot at first pool creation" check. The RugWatch repos (#5 and #8, independently) implement a launch-time scan: is initial liquidity sufficient? Is LP unlocked from genesis? These are static-state checks at a specific temporal point (pool Initialize event), not ongoing change detectors. This is a distinct detector slot from D01–D07.

**What it detects:** Tokens that are structurally rigged at birth — not from evolution but from design. Initial liquidity below a floor (e.g., <5 SOL, approximately $750 at current prices) means the pool has no economic moat against a single large sell. LP unlocked from genesis means the deployer can pull all liquidity in one transaction immediately after the first buys.

**Signal definition (precise):**
- Trigger: first `InitializePool` or `Initialize2` event for a given token
- Signal A: `initial_liquidity_sol < initial_liquidity_floor_sol` (proposed threshold: 5.0 SOL, from RugWatch consensus; needs calibration against our 100-token positive fixture)
- Signal B: `lp_locked_pct < lp_safe_floor_pct` at genesis (proposed: 0.0% locked = max risk; threshold from Alhaidari et al. 2025 SolRPDS Table 3 ≥70% as safe floor — already in REFERENCES.md)
- Confidence formula: weighted combination; both signals firing → conf ≥ 0.80
- Output: `AnomalyEvent { detector: "D08_launch_audit", confidence, severity: High, evidence: { initial_liquidity_sol, lp_locked_pct } }`

**Consumer value:** Trading bot — block trading on newly launched tokens that fail the launch audit before any volume accumulates. Custody — refuse to hold tokens that never had legitimate liquidity.

**Effort estimate:** S/M (2–4 hours). No new crate required. New file `crates/detectors/src/d08_launch_audit.rs`. Triggered by `PoolEvent::Initialize` in the existing streaming scheduler. Re-uses `MarketInfo.lockers` from D02 Signal B. Thresholds in `config/detectors.toml`.

**Prior art / citations:**
- RugWatch (machenxi + rookiester, both 2024–2025): <5 SOL initial liquidity as risk threshold — two independent implementations
- Alhaidari et al. 2025 (SolRPDS) Table 3: ≥70% LP burned/locked as safe floor — already in REFERENCES.md, directly applicable to Signal B at genesis
- Sun et al. 2024 §4 "Fake LP Lock" taxonomy: LP unlock as rug root-cause category — already in REFERENCES.md

**Risk:** Low. LP lock status query is already wired in D02. Initial liquidity is derivable from the Initialize event's reserve fields (already decoded in Raydium v4 + CPMM decoders). Threshold calibration needed against our 100-positive fixture (Sprint 4 P4-4 corpus includes launch-time data).

#### T1-2: Pump.fun bonding-curve graduation stream integration

**Gap description:** RugWatch repos (#5, #8) both explicitly target Pump.fun alongside Raydium/Meteora. Pump.fun uses a bonding-curve mechanism: tokens start on Pump.fun's curve and "graduate" to Raydium when they reach ~$69K market cap. The graduation event (a specific Pump.fun program instruction) is a high-signal moment — a large fraction of rug pulls happen within minutes of graduation when the bonding-curve liquidity is migrated to Raydium and the deployer can drain it. We have a Raydium adapter but no Pump.fun program instruction decoder.

**What it detects:** Not a new anomaly class — graduation itself is not suspicious. But graduating tokens arrive at Raydium with a known provenance: we know the Pump.fun deployer address, the bonding-curve price history, and the graduation timestamp. This metadata enriches D01–D07 immediately: a token that graduated 3 minutes ago and is already showing D02 LP drain signals is at 10× the risk of an older token showing the same drain percentage.

**Signal definition (precise):**
- Not a standalone detector — a metadata enrichment for existing detectors
- `TokenMeta` gains: `pump_fun_graduation_time: Option<DateTime>`, `pump_fun_deployer: Option<Address>`, `bonding_curve_price_at_graduation: Option<Decimal>`
- D02, D04, D05, D06 all gain a `time_since_graduation_minutes` evidence field
- The scoring crate gets a `graduation_recency_multiplier` config param: tokens < N minutes post-graduation get a confidence amplifier on any signal

**Consumer value:** Trading bot — most impactful. The bot can immediately flag "this is a brand-new graduate" and apply higher caution. Custody — refuse deposits of sub-5-minute-old Pump.fun graduates.

**Effort estimate:** M (3–5 hours). Need to decode Pump.fun program graduation instruction (the `migrate` or equivalent CPI into Raydium's Initialize). New migration for `token_metadata.pump_fun_graduation_time` column. No new detector file — enrichment only.

**Prior art / citations:**
- RugWatch (#5, #8): Pump.fun as explicit monitoring target — two independent production implementations
- Chainalysis 2025 (already in REFERENCES.md): "94% of rugged tokens have pool deployer as rug actor"; Pump.fun graduates are deployer-controlled by construction
- Shoaei et al. 2026 LROO (already in REFERENCES.md): "zero liquidity + zero tx + undefined price ≥72h → dead token; >95% within 1–3 days" — graduation window is the highest-risk early period

**Risk:** Medium. Pump.fun program IDs and instruction layouts are public but not yet decoded in our codebase. Pump.fun has upgraded their program at least twice (classic vs. AMM version). Need to verify current program ID and instruction discriminator. No archive-RPC cost: graduation events are sparse (a few hundred per day), so one-pass filtering of the Yellowstone stream is sufficient.

### Tier 2 — Next 2 sprints

#### T2-1: Bayesian changepoint detection on deployer behavior (D-BPCD, Phase 3 graph prereq)

**Gap description:** Latent-flux (#10) implements Adams & MacKay (2007) BOCPD on transaction sequences. The key insight: a deployer who has previously launched 10 legitimate tokens and then suddenly changes behavior (faster drain cadence, different wallet clustering pattern) is detectable via changepoint detection even if no single individual signal breaches a threshold. This is the temporal dimension missing from our static deployer_clusters table.

**Signal definition (precise):**
- Input: per-deployer time-series of: token_count_launched, time_between_launches, mean_lp_locked_pct, mean_holder_count_at_launch, fraction_of_tokens_that_rugged
- Changepoint: Adams & MacKay 2007 online BOCPD with Normal-Gamma conjugate prior; hazard rate 1/300 events (approximate; calibrate on our positive fixture corpus)
- Output: `deployer_risk_score: Decimal` that updates continuously; step-change in score triggers an `AnomalyEvent { detector: "DBPCD_deployer_changepoint" }`
- Dependency: requires `deployer_clusters` table population (Phase 3 graph)

**Consumer value:** All four consumers. Allows pre-screening of deployer addresses before a new token from that deployer is even traded.

**Effort estimate:** L–XL (1–2 sprints). Requires Phase 3 graph work first (deployer edge computation). The BOCPD algorithm itself is straightforward (pure Rust, no ML framework). But the data pipeline (indexing deployer history, maintaining per-deployer time-series in Postgres) is the work.

**Prior art:** Adams & MacKay 2007 (online BOCPD) — peer-reviewed, well-cited; latent-flux production implementation confirms feasibility; Chainalysis 2025 (deployer behavior as leading risk signal, already in REFERENCES.md).

**Risk:** Medium. BOCPD is well-understood but parameter sensitivity (hazard rate, prior hyperparameters) requires calibration against our positive fixture corpus. Dependency on Phase 3 graph crate is a blocker.

#### T2-2: Graph cycle detection for wash-trading (Tarjan SCC + Johnson)

**Gap description:** Latent-flux (#10) uses Tarjan SCC + Johnson's cycle enumeration on the address transfer graph to detect circular fund flows (A→B→C→A). This is mathematically stronger than our D05 Heuristic 1 (same-address round trips) and Signal B (cluster flow balance). A wash ring of 3+ distinct wallets cycling tokens is invisible to D05 H1 but immediately caught by cycle detection on the graph.

This is the Phase 3 "D05 graph-backed Signal B" already registered in our ROADMAP.

**Signal definition (precise):**
- Input: transfer edges in the `graph/` crate (Phase 3 prerequisite)
- Tarjan SCC pre-filters: identify strongly connected components in the transfer graph (O(V+E))
- Johnson's algorithm enumerates all elementary cycles within SCCs of size ≥ 3
- Signal: cycle_length ≤ max_cycle_length (proposed: 5 hops), cycle_volume_usd ≥ min_cycle_volume_usd (proposed: $1,000), cycle_time_window_minutes ≤ 120
- Output: amplifier/evidence for D05 (replaces Signal B proxy)

**Effort estimate:** L (1 sprint). Tarjan + Johnson are well-known algorithms with Rust crate implementations (e.g., `petgraph`). Integration into Phase 3 graph crate is the work.

**Prior art:** Tarjan 1972 (foundational graph algorithm); Victor & Weintraud 2021 (circular trade patterns — already in REFERENCES.md); latent-flux (#10) production implementation.

**Risk:** Medium. Depends on Phase 3 graph crate being populated with transfer edges. Without the graph the algorithm has no input. O(V+E) SCC is fine; Johnson's worst-case is O((V+E)×C) where C = number of cycles — need cycle count ceiling to avoid runaway on large clusters.

#### T2-3: Funding rate + open interest context layer (Phase 4 EVM / CEX-integrated)

**Gap description:** Vizzor (#9) ingests Binance Futures funding rates and open interest as signals that precede or accompany pump events. A spike in funding rate (longs paying shorts heavily = crowded long = price likely to revert or dump) combined with rising OI and rising price is a precursor to a leveraged dump. This is not an on-chain signal — it comes from CEX derivatives data. But it is actionable context for the trading bot: if our D04 pump signal fires AND external funding is at extreme positive levels, confidence should be higher.

**Signal definition (precise):**
- Not a standalone detector — a context enrichment for D04 (Pump & Dump)
- New data source: Binance Futures public API (funding rate endpoint, `/fapi/v1/fundingRate`) — self-hostable polling, no SaaS dependency, public unauthenticated endpoint
- D04 gains an optional `funding_rate_context: Option<FundingRateContext>` field in evidence
- Scoring crate gets a `funding_rate_amplifier` config param (proposed: +0.10 confidence when funding > 2× 30-day mean)
- Dependency: tokens must have a corresponding Binance perpetual futures contract (only applies to larger tokens; BONK, WIF, etc.)

**Consumer value:** Trading bot — highest impact. Market maker — position risk context.

**Effort estimate:** M (3–4 hours for data fetcher + D04 integration). The Binance Futures public REST API is unauthenticated. A lightweight async fetcher caching funding rates by symbol.

**Prior art:** Vizzor (#9) production implementation; academic literature on funding rate as manipulation signal is thin but industry consensus is strong (Binance, FTX derivatives manipulation playbooks documented in numerous post-mortems).

**Risk:** Low-Medium. ADR 0003 ("self-sovereign infrastructure") — Binance public REST API is a read-only unauthenticated endpoint, not a paid SaaS dependency. Acceptable. Risk: only applicable to tokens that have a perp contract (reduces coverage to top ~200 Solana tokens). For long-tail shitcoins (our primary target), no futures data available — must be an optional enrichment only.

#### T3-2: Stablecoin depeg detection via liquidity curve PCA (EVM Phase 4)

**Source:** QuantLet/onchain-insights (#2, published paper)
